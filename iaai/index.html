Business Analytics Overview
Foundation ▾

What is Business Analytics?

The systematic use of data, statistical and quantitative analysis, explanatory and predictive models, and fact-based management to drive decisions and actions.

  • Descriptive: What happened? (dashboards, reports)
  • Predictive: What will happen? (ML models, forecasting)
  • Prescriptive: What should we do? (optimization, decision support)
Key Principle Data-Driven Decision Making (DDD) increases productivity by 5–6% (Brynjolfsson et al., 2011). Companies that use data analytically outperform peers consistently.

Why AI Now?

  • Exponential growth in data volume (IoT, social, transactions)
  • Dramatic drop in compute costs (cloud, GPUs)
  • Algorithmic breakthroughs (deep learning, transformers)
  • Availability of pre-trained models and APIs

AI Profitability Reality
▾

Not all AI investments generate returns. The profitability of AI depends on where it fits in the competitive landscape and whether the firm has the data and talent to execute.

  • Most value: AI embedded in core business processes with proprietary data
  • Least value: Generic AI tools applied without business context
  • Key risk: Treating AI as a magic solution vs. a specific tool for specific problems
Reality Check Most AI projects fail not because of algorithms, but because of poor problem formulation, insufficient labeled data, or misalignment with business goals — reinforcing the importance of CRISP-DM.

AI System Architecture
▾

Real AI systems are pipelines, not single models. A production AI system typically contains: data ingestion layer → feature engineering → one or more ML models → decision logic → output/action layer.

Input Instance for an LLM The input instance for an LLM is the prompt. Everything the model sees — instructions, context, prior conversation — is the input instance it predicts from.
  • Data pipeline: Collect, clean, transform features
  • Model layer: One or more trained models producing scores/predictions
  • Decision logic: Business rules that act on model outputs
  • Feedback loop: Outcomes feed back to improve future models

Mega Bank — Customer Targeting Case
Case Study ★ ▾

A bank wants to identify which customers to target with a premium credit card offer. Classic supervised learning formulation.

Problem Formulation

  • Target variable: Will customer accept premium card offer? (Yes/No)
  • Features: Transaction history, demographics, balance, tenure
  • Objective: Maximize expected profit from campaign

Key Lesson: Multiple Comparisons Problem

Running many statistical tests and reporting only the "winners" inflates apparent significance. If you test 1,000 features, ~50 will look significant at p<0.05 purely by chance. This is a form of overfitting to the data's noise.

Business Implication This is why holdout evaluation matters: results that look good on training data may be spurious. Always verify on held-out data before deploying.
🎲
Multiple Comparisons & Overfitting
▾

Overfitting = tendency to tailor models to training data at the expense of generalization to new data. (DSB Ch. 5)

  • A table model (memorizing all training examples) achieves 100% training accuracy but fails completely on new data
  • As model complexity increases: training error falls, but holdout error eventually rises
  • "If you torture the data long enough, it will confess." — Ronald Coase

Complexity Control

  • Tree pruning: Stop growing before too many nodes
  • Cross-validation: k-fold CV gives better estimate of true generalization
  • Regularization: Penalize complexity in the objective function (L1/Lasso, L2/Ridge)
  • Feature selection: Remove spurious features before modeling
Fitting Curve vs Learning Curve Fitting curve: x = model complexity, y = error (training + holdout). Learning curve: x = training data size, y = generalization accuracy. Both help diagnose underfitting vs overfitting.
🌳
Tree Models & Complexity
▾

Classification trees divide the feature space into regions using axis-parallel splits. Each split maximizes information gain (reduction in entropy).

Entropy(S) = -p₊ log₂(p₊) - p₋ log₂(p₋)
InfoGain(S, A) = Entropy(S) - Σ (|Sᵥ|/|S|) · Entropy(Sᵥ)

Key Properties

  • Interpretable: can be expressed as IF-THEN rules
  • Axis-parallel decision boundaries (not diagonal)
  • High variance: different training sets → very different trees → reason for ensemble methods (Random Forests)
  • Laplace correction for probability estimation at leaves: p(c) = (n+1)/(n+m+2)

Ensemble Methods

  • Random Forest: Build many trees (bagging), each uses random √p feature subset, then majority vote
  • Reduces variance by averaging diverse trees
  • Bagging (bootstrap aggregating) + random feature subsets = double source of diversity

Model Evaluation Methods
▾

Holdout Evaluation

Set aside data not used to train the model. Evaluate on holdout to estimate generalization. Simple but wastes data.

k-Fold Cross-Validation

Split data into k folds. Train on k-1, test on 1. Rotate k times. Average accuracy across folds. Standard: k=5 or k=10.

Why CV beats single holdout CV gives k accuracy estimates → compute mean and standard deviation. Better use of limited data, more reliable estimate of true generalization error.

Nested Cross-Validation

For hyperparameter tuning: outer loop for evaluation, inner loop to select complexity parameter. Prevents information leakage from test fold.

Classification Framework & Expected Value
▾

When making targeting decisions, use the Expected Value framework rather than raw accuracy.

Expected Profit = Σ p(outcome) × value(outcome)

Cost-Benefit Matrix

Each cell in the confusion matrix has a business cost/benefit:

OutcomeBusiness Value
True Positive (TP)Revenue − cost of offer
False Positive (FP)−cost of offer (wasted)
False Negative (FN)−lost opportunity (customer not reached)
True Negative (TN)$0 (no action taken)
Key Insight Accuracy is almost always the wrong metric for real business problems. The expected value framework forces you to connect model performance to dollars.
📈
ROC Analysis & AUC
▾

ROC (Receiver Operating Characteristic) curves plot True Positive Rate vs False Positive Rate as you sweep the classification threshold. They are independent of class priors.

  • x-axis: False Positive Rate (FPR = FP/(FP+TN))
  • y-axis: True Positive Rate (TPR = TP/(TP+FN))
  • Diagonal: Random classifier performance
  • Upper-left corner: Perfect classifier

AUC — Area Under the Curve

  • AUC = 0.5: random; AUC = 1.0: perfect
  • Interpretation: probability that a random positive instance ranks higher than a random negative
  • Best single-number summary when operating conditions are unknown

Lift Curves & Cumulative Response

Lift = CRC value / diagonal value at same targeting %. A 2× lift means you're twice as efficient as random targeting. Lift curves are class-prior dependent; ROC curves are not.

⚖️
Threshold Selection Strategies
▾

Rather than defaulting to 0.5, the optimal threshold comes from the cost-benefit tradeoff:

Target customer if: p(buy|x) > Cost / (Revenue − Cost)
Example: product costs $1 to mail, profit if bought = $99
Threshold = 1/(99+1) = 1% — target anyone with >1% probability!
Key Takeaway The 50% threshold is almost never optimal. In many business settings (offers, direct mail), the threshold is far lower because the cost of outreach is small relative to the potential gain.

Budget-Constrained Decisions

With a fixed budget, use profit curves to find the optimal targeting percentage. The threshold becomes: how many top-ranked customers can we afford to target?

Model Selection Paradox
Case Study ★ ▾

The most accurate model is not always the best business choice. This paradox arises when class distributions differ between evaluation and deployment environments.

KDD Cup Churn Example

ModelTraining AccuracyCV AccuracyAUC
Classification Tree95%91.8%0.614
Logistic Regression93%93.0%0.574
k-NN100% (overfit!)93.0%0.537

Paradox Naive Bayes has the LOWEST accuracy but the HIGHEST AUC. On imbalanced data (7% churn rate), accuracy favors models that simply predict "no churn" for everyone. AUC reveals the true ranking quality.

Digital Advertising — Distillery Case
Case Study ★ ▾

Display advertising requires predicting whether a user will click or convert on an ad in real-time (milliseconds). The Distillery case shows how data science creates competitive advantage in programmatic advertising.

Key Concepts

  • RTB (Real-Time Bidding): Auction for each impression, <100ms to decide bid
  • CTR Prediction: p(click | user, ad, context) — the core ML problem
  • Targeting features: Browsing history, demographics, time of day, device, site context
  • Revenue model: CPM (cost per thousand impressions) or CPC (cost per click)

Lessons Learned

  • Low base rates (0.1–0.3% CTR) make accuracy misleading — use AUC and lift
  • Data volume matters: billions of impressions → need scalable models
  • Feature freshness: user intent decays rapidly (minutes, not days)
  • The real competitive advantage is the data asset + feedback loop, not any single algorithm

Required Reading ▾

Kosinski, Stillwell & Graepel (2013). "Private traits and attributes are predictable from digital records of human behavior." PNAS.

Main Finding

Facebook Likes alone can accurately predict highly sensitive personal attributes — without users knowingly disclosing them.

Methodology

  • Method: SVD (dimensionality reduction) → Logistic/Linear Regression → 10-fold CV

Data Science Problem Formulation — Targeting Offers
Foundation ▾

Labeled Data Challenges: Business vs. Academic

ContextReality
AcademicCan abandon project if labeled data unavailable; pivot to different dataset
BusinessMust find a solution; data gaps must be addressed within business understanding phase

Five Approaches When Lacking Labeled Data

  • Use a different offer type targeting the same customer segments
  • Estimate probability of staying based on comparable historical data
  • Assumption: similar people respond similarly to related offers (proxy modeling)
  • Use control groups in new campaigns to generate labels going forward
  • Invest in a small-scale pilot with random assignment to generate ground truth

Expected Loss Targeting

More sophisticated than churn probability alone:

Priority Score = P(leaving) × Customer Lifetime Value (CLV)
  • Budget allocation prioritizes high-value customers within the same risk tier
  • Assumption (when no other data): offer effectiveness = 100%
  • Better approach: estimate causal lift (see AI System Architecture section)
Key Insight Don't target based on churn probability alone. A low-value customer who's definitely leaving is less worth targeting than a high-value customer with moderate churn risk.

AI System Architecture — Multiple Models & Decision Logic
▾

Modern AI systems are not a single large model. They are pipelines of specialized models connected by decision logic.

Two-Model Churn Prevention System

ModelOutputUse
Model 1P(staying | targeted with offer)Probability customer stays if we intervene
Model 2P(staying | not targeted)Probability customer stays without intervention
Decision LogicΔp × CLV − cost > 0?Target only if net expected value is positive

Decision Logic Inputs Beyond Model Outputs

  • Cost of the offer being considered
  • Customer Lifetime Value (CLV) — computed by a separate system
  • Available budget constraints
  • Cost matrices for different error outcomes

Counterfactual Causal Framework

Core Concept Causal effect = P(staying | targeted) − P(staying | not targeted). We compare "what if we target" vs "what if we don't." The fundamental challenge: we cannot observe both outcomes for the same customer simultaneously. We estimate average treatment effect across customer groups.
Causal Effect = P(Stay | T=1) − P(Stay | T=0)

This is why A/B testing is powerful — randomization allows valid comparison of both counterfactuals.

LLM Technical Foundation — Tokens & Probability Distributions
Technical ▾

What Is a Token?

  • Words (majority of tokens)
  • Punctuation marks and spaces
  • Coding symbols and special characters
  • Word pieces — common syllables, prefixes, suffixes (e.g., "un-", "-ing")
  • Individual letters as fallback for unknown terms
  • Total vocabulary size: ~20,000–30,000 tokens

LLM Output: Distribution, Not a Single Answer

  • Model outputs scores for all ~30,000 possible next tokens simultaneously
  • Scores converted to probability-like distribution (not well-calibrated as true probabilities)
  • Decision logic samples from high-scoring tokens — not always the highest-scoring one
Temperature Parameter Controls randomness of sampling. Temperature = 0: always pick the highest-scoring token (deterministic). Higher temperature: more random, more creative but potentially less accurate. Most production LLMs use temperature > 0.

Generation Process

StepWhat Happens
InputSparse representation — indices of present tokens only
Forward passModel outputs probability scores across full vocabulary
SampleSelect next token based on temperature-modulated distribution
AppendSelected token appended to input; cycle repeats
StopEnd-of-sequence (EOS) token triggers completion

Tokenization vs Chunking (RAG)

  • Tokenization: Breaking input into model-ready features for inference
  • Chunking: Semantic document segmentation for retrieval (RAG pipelines)
  • Both serve different purposes and operate at different stages of the AI pipeline

LLM Training & Capabilities — Beyond Next Word Prediction
▾

Pre-Training as Supervised Learning

  • Training data: Curated internet text (not random web scraping)
  • Instance: Any text substring
  • Label: The actual next token that follows in the original text
  • Requires vast knowledge to predict accurately across all topics → forces deep generalization

Instruction Following & RLHF

  • Human feedback on answer quality ("which response do you prefer?")
  • Professional raters evaluate responses at scale (Reinforcement Learning from Human Feedback)
  • Trains for task completion and helpfulness, not just text continuation

Knowledge Storage & Generalization

  • Models store facts, relationships, and reasoning patterns in weights
  • Can modify inferences when facts are provided in context
  • Main value is generalization to unseen inputs, not memorization
  • Example: writing rhyming sonnets requires long-term planning beyond single next-token
Probability Calibration Note LLM output scores are not true probabilities of real-world events. They're more like "relevance weights." Treat confidence levels with skepticism — LLMs can be confidently wrong.

Token Costs & Efficiency

  • Pricing is per input + output token
  • Large document inputs = higher charges
  • Use RAG to retrieve relevant chunks rather than including entire documents in context

Modern AI System Components — Memory, RAG & Tool Integration
Architecture ▾

Short-Term Memory for Multi-Step Planning

  • System can write plans ("I'll do A, then B, then C") within context window
  • Access memory to track progress and resume complex tasks
  • Enables workflows spanning many steps beyond single-turn responses

RAG — Retrieval Augmented Generation

  • System retrieves relevant document chunks semantically related to the query
  • Chunks incorporated into prompt as context
  • Reduces reliance on memorized training data for factual questions
  • Example: web search integration fetches current information

Tool Calling

  • Calculator: LLMs are poor at complex arithmetic — offload to tools
  • Code execution: Run Python/R for computational tasks
  • API calls: Query external services, databases, web
  • Mechanism: LLM generates tool call text → executed → result fed back as next prompt → repeat

A/B Testing Framework — Playlist Recommendation Case (Songify)
Case Study ★ ▾

Unit of Analysis

  • Users who access personalized playlist features (not all users)
  • Exclude: users who never visit discovery pages
  • Consider segmentation: free vs. premium users separately
  • Ensure consistent user experience across sessions (sticky assignment)

Hypothesis

New ML playlist generator increases user engagement, measured by streaming time from generated playlists, by at least X% with statistical significance.

OEC — Overall Evaluation Criterion

MetricStatusReason
Total streaming time from playlists✓ ChosenMeasurable across all users, correlates with engagement, simple
Song skips✗ RejectedHard to normalize; ambiguous signal
Saves / Downloads✗ RejectedNot all users take these actions
Premium upgrades✗ RejectedToo distal; confounded by other factors
Multiple metrics✗ RejectedConflicting results create ambiguity

Statistical Framework

  • Randomization: Equal probability assignment to treatment/control groups
  • Balance checks: Verify similar demographics post-randomization (sanity check)
  • One-tailed test: Appropriate when testing "is new better?" (directional hypothesis)
  • 95% CI standard: But practical significance matters more than statistical significance

Sample Size & Power Analysis

  • Determine Minimum Detectable Effect (MDE) — practical significance threshold
  • Calculate required N for 80–90% statistical power
  • Balance test duration vs. sample size needs
  • Account for seasonal effects and user lifecycle timing
Avoid Multiple Metrics Pick one OEC before running the test. Checking multiple metrics post-hoc reintroduces the multiple comparisons problem — inflating false discovery rate.

Competitive Advantage — AdSafe / Integral Ad Science Case
Case Study ★ ▾

Problem Context

  • Complex ad tech ecosystem with 200+ players
  • Obfuscation: Publishers hide identity to protect premium ad sales
  • Ad networks aggregate small sites for scale; contract trading creates opacity
  • Brand ads appearing next to objectionable content (violence, hate speech, adult)

Technical Solution — Reverse Firewall

ComponentDescription
Crawling & ScoringOffline web crawling; ML models score pages/sitelets across ~24 content categories
Real-time lookupDatabase lookup during ad serving — millisecond latency
ConfigurabilityBrand-specific threshold settings per content category
Error handling99% accuracy target; blacklist for missed content; human review team

Operational Challenges

  • Crawl lag: New pages may receive ads before scoring completes
  • False positives: Satire detection difficulties; comedy site whitelisting needed
  • Pricing: CPM add-on (~$0.05 per thousand impressions); scales naturally with usage

Five Competitive Sustainability Factors (DSB Ch. 13 Applied)

  • 1. Intellectual Property: Patents on ad blocking methods; trade secret algorithms
  • 2. Switching costs: Integration complexity + threshold recalibration effort
  • 3. Low marginal costs: ~100% margin enables aggressive price defense
  • 4. Publisher partnerships: Revenue sharing for site cleanup creates aligned incentives
  • 5. First-mover network effects: Established customer relationships and references

Real Options Strategy

  • Core data asset: Visibility into every ad impression across the internet
  • Expansion products: Ad fraud detection, viewability measurement, campaign analytics
  • Revenue: $100M+ annual revenue by 2016 from single infrastructure investment
  • Strategic pivot: Brand safety became secondary to fraud/viewability as market evolved
Key Lesson The data moat (seeing every impression) enabled multiple product lines. The first use case (brand safety) was a beachhead; the real value emerged from the data asset's generality.

Iterative Development — Spiral Investment Model
▾

Core Principle: Staged Investment

  • Start with small proof-of-concept to test core assumptions
  • Increase investment only after reducing key uncertainties
  • Each iteration is larger scope but lower risk than the last

Pilot Design Principles

  • Use existing data rather than collecting new datasets
  • Focus on a single use case rather than comprehensive solution
  • Measure generalization error on holdout data (not training performance)
  • Get stakeholder feedback on model outputs and interface quality

Power Analysis Alignment

  • Sample size calculations tied to practical significance thresholds (not just statistical)
  • Test duration balances statistical power with business timeline needs
  • Account for seasonal effects and user behavior cycles

Result: $500M Revenue Increase in Year 1 Workshop process: ML estimates + expert validation. 50% accepted model recommendations directly. 40% modified based on local knowledge. 10% set opportunities to zero (no viable prospects). 10% sales force reallocation drove the incremental result — the model's value came from augmenting human judgment, not replacing it.

Day 1 Core Takeaways

1
AUC beats Accuracy. On imbalanced data, a model with 76% accuracy can outperform one with 93% accuracy in terms of ranking quality (AUC 0.632 vs 0.537). Always pick the metric tied to the business goal.
2
Threshold is a business decision. The 50% threshold is almost never optimal. When mailing cost = $1 and potential profit = $99, the break-even threshold is just 1%.
3
CRISP-DM is iterative, not linear. Most real projects cycle back from Modeling to Data Understanding multiple times. The process is a spiral, not a waterfall.
4
Multiple comparisons = overfitting. Testing 1,000 features finds ~50 "significant" ones by chance. Holdout evaluation is the only defense.
5
Proxy features reveal everything. Kosinski (2013): Facebook Likes predict gender (93% AUC), ethnicity (95%), political views (85%). The act of clicking Like is the signal — not the underlying preference.
6
Expected value, not accuracy. Use cost-benefit matrix × confusion matrix probabilities to compute expected profit per customer. This is the right objective for business decisions.

Day 2 Core Takeaways

1
Target by expected loss, not churn probability. P(leaving) × CLV is more powerful than P(leaving) alone. A low-value churner is less worth targeting than a high-value moderate-risk customer.
2
Causal effect, not correlation. The counterfactual framework: target those where P(Stay|T=1) − P(Stay|T=0) is highest AND CLV is high. Without random assignment, causal claims are suspect.
3
LLMs output distributions, not answers. The model scores all ~30,000 tokens. Temperature controls how deterministic the sampling is. The output is never a single determined answer.
4
Data uncertainty kills AI projects. Of the 5 uncertainty types, data access/quality is the most common failure mode. Audit data before building models.
5
One OEC, chosen before the test. Pick a single Overall Evaluation Criterion for A/B tests. Post-hoc metric selection reintroduces the multiple comparisons problem.
6
Data moat = multiple product lines. AdSafe/IAS's visibility into every impression enabled fraud detection, viewability, and brand safety from a single infrastructure investment. The beachhead product is rarely the real value.

Question 1 · Day 1 · Model Evaluation
In the KDD Cup churn dataset, Naive Bayes had the lowest accuracy (76%) but the highest AUC (0.632). Why does this happen, and which metric should you trust?
AUC measures the probability that a random positive instance ranks higher than a random negative — a class-prior-independent measure of ranking quality. On heavily imbalanced data, a model predicting "no churn" for everyone gets 93% accuracy but adds zero business value. Naive Bayes, despite lower accuracy, actually identifies churners and therefore has higher AUC and real predictive utility.
Question 2 · Day 1 · Threshold Selection
A retailer sends a $2 direct mail piece. A customer who responds buys on average $50 of product with $40 margin. Using the threshold formula, what is the minimum response probability that justifies mailing?
The formula is: p(response) > cost / (benefit − cost) = $2 / ($40 − $2) = 2/38 ≈ 5.3%. Mail anyone with a predicted response probability above ~5%. The 50% default is almost always wrong in direct marketing settings where costs are low relative to gains.
Question 3 · Day 1 · Overfitting
What is the "table model" thought experiment, and what does it demonstrate about overfitting?
The table model is a thought experiment from DSB Ch. 5: a "model" that simply stores every training example and looks up the answer for exact matches. It achieves 100% training accuracy but cannot generalize to any new instance not in training data — the extreme case of overfitting. It illustrates why training accuracy alone is meaningless.
Question 4 · Day 1 · Kosinski Reading
In Kosinski et al. (2013), what was the AUC for predicting sexual orientation (male) from Facebook Likes, and what does this finding imply for privacy?
AUC 0.88 for sexual orientation prediction from Likes. The privacy implication is profound: users clicking "Like" on innocuous pages (Lord of the Rings, etc.) are inadvertently revealing sensitive attributes they never chose to disclose. This is the proxy feature problem — behavioral data predicts protected attributes that companies couldn't legally ask about directly.
Question 5 · Day 2 · LLM Architecture
What does the temperature parameter control in an LLM, and what happens at temperature = 0?
Temperature is applied to the logit scores before sampling: low temperature sharpens the distribution (near-deterministic), high temperature flattens it (more random). T=0 = always pick the argmax token. This matters for product design: creative tasks benefit from higher temperature; factual Q&A benefits from lower temperature.
Question 6 · Day 2 · Counterfactual Framework
Why is it problematic to target churn prevention offers to customers with the highest churn probability alone?
The counterfactual framework shows that the right target is: high Δp (the offer actually changes P(staying)) × high CLV. Some customers will churn regardless of any offer — "certain churners" — and targeting them wastes budget. Others would have stayed anyway — "certain stayers" — and you're giving away margin unnecessarily. You want customers in the "persuadable" zone where the offer genuinely moves the needle.
Question 7 · Day 2 · A/B Testing
In the Songify playlist experiment, why was "total streaming time from generated playlists" chosen as the OEC over metrics like song saves or premium upgrades?
A good OEC must be: (1) measurable for all users in the experiment, (2) directly tied to the hypothesis, and (3) simple enough to interpret unambiguously. Streaming time passes all three. Saves/downloads require user action beyond streaming (many won't take them), premium upgrades are too distal and confounded, and multiple metrics create conflicting signals and a multiple comparisons problem.
Question 8 · Day 2 · AI Project Management
Of the five sources of AI project uncertainty, which is most commonly cited as the primary failure mode in enterprise AI, and why?
Data uncertainty is the #1 killer of enterprise AI projects. It encompasses: data silos (can't get access), quality problems (inconsistent, outdated, mislabeled data), integration complexity (multiple formats/systems), and political resistance from data owners. The algorithm is rarely the bottleneck — it's getting clean, labeled, accessible data at the right granularity.
Question 9 · Day 2 · Competitive Advantage
AdSafe/IAS's core competitive advantage came from what strategic asset, and how did it enable expansion beyond brand safety?
The "data moat" — seeing every impression — was the core asset. Brand safety was just the entry point (beachhead product). Because IAS already processed every impression for content scoring, adding fraud detection and viewability measurement required marginal additional cost but created entirely new revenue streams. This is the "real options" strategy from DSB Ch. 13: invest in a data asset that enables future products you may not have planned for upfront.
Question 10 · Both Days · Synthesis
The Zillow Offers failure combined which two concepts covered across Day 1 and Day 2?
Two compounding failures: (1) Selection bias — Zillow only observed houses where sellers accepted offers, meaning training data systematically under-represented overpriced homes. Homeowners rationally accept low-ball offers only when they believe their home is actually worth less, so the training set was biased toward "algorithm already overpaid" scenarios. (2) Market reaction — the model assumed a static environment, but sellers and agents adapted, making the model's assumptions progressively less valid after deployment. Day 1 concept (selection bias / data understanding) + Day 2 concept (anticipating ecosystem reactions to AI deployment).
Question 11 · Ch. 8 · Model Performance Visualization
A lift chart shows your model scores 3.2× at the top decile. What does this mean operationally?
Lift at a given percentile = (response rate in that bucket) / (overall response rate). A lift of 3.2× at the top decile means that if you mail only the top 10% of customers ranked by model score, you get 3.2 times as many responses per person as if you mailed randomly. This directly translates to ROI: same budget, 3.2× the hits. The lift curve is the practitioner's tool for deciding how deep to cut into a ranked list.
Question 12 · Ch. 7 · Decision Analytic Thinking
In the DSB cost-benefit framework, expected profit from targeting a customer = ?
Expected profit = p(response) × benefit(TP) + [1−p(response)] × cost(FP). Concretely: if a customer has a 15% predicted response probability, the offer costs $2, and a responder generates $40 profit: Expected value = 0.15 × $40 + 0.85 × (−$2) = $6.00 − $1.70 = $4.30 → profitable to mail. The framework replaces "is this customer likely to respond?" with "is targeting this customer profitable in expectation?" — a fundamentally different and more useful question.
Question 13 · Ch. 11 · Profit and Competition
In the churn counterfactual framework, who should you NOT target with a retention offer, even if they have a high churn probability?
The four segments: (1) Certain stayers — waste of offer budget, they stay regardless. (2) Certain churners — waste of offer budget, they leave regardless. (3) Persuadables — high Δp (the offer genuinely moves P(stay)), this is your target. (4) Do-not-disturbs — the offer might actually cause them to churn (reactance). The business variable is Δp × CLV, not just P(churn). A customer with 90% churn probability but Δp ≈ 0 is not worth targeting.
Question 14 · Ch. 13 · Data Science Strategy
What is a "data moat" and which case study from class best illustrates it?
A data moat is a self-reinforcing competitive advantage built on data that grows harder to replicate over time. IAS (formerly AdSafe) was the canonical example: by processing every ad impression for brand safety, they accumulated a scale of labeled impression data no competitor could match. This data moat enabled launching new product lines (fraud detection, viewability) at near-zero marginal cost. DSB Ch. 13 frames this as "real options" — the initial investment buys not just the current product but the option to build future products from the same data infrastructure.
Question 15 · Ch. 13 · Data Network Effects
What is a data network effect and why does it create durable competitive advantage?
The data network effect loop: more users generate more behavioral data → data trains better predictive models → better models improve the product experience → better product attracts more users → cycle repeats. Each iteration widens the gap vs. late entrants. Examples: Google Search (more queries → better ranking algorithms → users prefer Google → more queries), Facebook (more profiles → better ad targeting → more advertisers → better content/reach → more profiles). A new entrant with the same algorithm but without historical data cannot replicate the performance, regardless of engineering talent.
Question 16 · Ch. 13 · Real Options
In DSB's real options framework, why might a firm collect data even if it has no immediate use for it?
Real options in finance = the right (not obligation) to make a future investment. Applied to data: collecting behavioral data today buys you the option to build a model or answer a question in the future — even if that question hasn't been asked yet. DSB Ch. 13 example: Harrah's (now Caesars) built a massive customer database for loyalty tracking. Years later, that same data enabled predictive CLV models, fraud detection, and personalization they never anticipated. The decision to collect data should be evaluated not just on known current use cases but on the option value for unknown future applications.
Question 17 · Ch. 14 · Interpretability vs. Accuracy
When is interpretability more important than predictive accuracy, and when is the reverse true?
DSB Ch. 14 frames this as context-dependent. Interpretability is critical when: (1) humans must review and act on model recommendations (they need to know why), (2) regulations require explainability (GDPR, lending decisions), (3) the model could embed bias that needs auditing. Accuracy matters more when: (1) the model acts autonomously with measurable outcomes (spam filter, fraud detection), (2) stakeholders only care about aggregate performance. The Mega Bank example illustrates the middle ground — a model ML estimated, but sales reps modified based on local knowledge. High interpretability enabled the human-in-the-loop collaboration that drove $500M in revenue.
Question 18 · Ch. 8 · Profit Curves
In a profit curve plotted against percentage of customers targeted (x-axis), what does the peak of the curve represent?
Profit curves plot cumulative expected profit as you move down the model's ranked list. The curve rises as long as each additional targeted customer contributes positive expected profit (P(respond) × benefit > cost). After the peak, you're targeting customers where cost exceeds expected benefit — profit declines. The peak = optimal cutoff. This replaces arbitrary thresholds (50%) with a business-grounded one derived from the actual cost-benefit structure. If you have a fixed mailing budget, you read off the x-axis percentage at the peak and target that fraction of your list.
Question 19 · Ch. 13 · Privacy and Proxy Features
Why could Facebook Likes predict sexual orientation (AUC 0.88) even though no Like directly encodes sexuality? What broader principle does this demonstrate?
The proxy feature problem: protected attributes (sexuality, race, political views) that companies cannot legally collect are nonetheless predictable from behavioral signals that correlate with them through social network structure. Liking "Wicked the Musical" or "Kathy Griffin" doesn't mention sexuality — but the aggregate pattern of Likes across millions of users reveals demographic clusters. Kosinski (2013) showed this for gender (93%), ethnicity (95%), political views (85%), IQ (r=0.39). Implication: even if a firm is "not collecting" sensitive attributes, behavioral data may encode them implicitly. This is a central tension in algorithmic fairness and data ethics (DSB Ch. 14).
Question 20 · Ch. 7–11 · Synthesis
A colleague argues: "Our model has 91% accuracy — we should deploy it immediately for loan decisions." What is the most complete and correct pushback?
Three-part pushback tying together the full IAAI framework: (1) Class imbalance — if 91% of loans are repaid, a model predicting "approve everyone" gets 91% accuracy but catches zero defaults. (2) Asymmetric costs — a missed default (false negative) costs the bank the full loan amount; a wrongly denied loan (false positive) costs only lost interest income. These are NOT equal, so the 50% threshold is wrong. (3) Expected profit > accuracy — the right framework is cost-benefit matrix × confusion matrix probabilities summed across all decisions. The optimal threshold comes from the profit curve, not a fixed cutoff. Additionally: loan decisions have legal interpretability requirements (ECOA, FCRA) — a black-box model may violate fair lending law regardless of accuracy.
